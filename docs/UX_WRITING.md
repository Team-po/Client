# UX Writing Guide

Team-po의 제품 문구 기준입니다. 사람이 직접 문구를 고칠 때도, AI가 문구 작업을 도울 때도 이 기준을 따릅니다.

Team-po는 개발자가 사이드 프로젝트 팀을 찾고, 팀 작업을 이어갈 수 있게 돕는 서비스입니다. 문구는 설명문이 아니라 제품 인터페이스처럼 느껴져야 합니다.

## 목소리

국내 SaaS 제품처럼 씁니다.

- 명확하게: 사용자가 무엇을 할 수 있는지 말합니다.
- 실용적으로: 시스템보다 사용자의 일을 중심에 둡니다.
- 차분하게: 과장, 압박, 장식적인 표현을 줄입니다.
- 구체적으로: 실제 화면과 동작에 맞는 단어를 씁니다.

도움말, 빈 상태, 안내 문구는 짧은 `-요` 문장을 기본으로 합니다.

## 원칙

### 1. 사용자 흐름부터 확인하기

문구를 고치기 전에 실제 흐름을 확인합니다.

- 사용자는 어느 화면에 있는가?
- 지금 어떤 상태인가?
- 누르면 무엇이 일어나는가?
- 로그인, 데이터, 설정 같은 선행 조건이 있는가?

좋은 문구는 사용자의 기대와 실제 동작이 어긋나지 않습니다. 중간 단계가 필요하다면 바로 접근할 수 있는 것처럼 쓰지 않습니다.

### 2. 다음 행동을 쓰기

대부분의 UI 문구는 사용자가 다음 행동을 정하게 도와야 합니다.

좋은 예:

- `역할을 선택해요`
- `제안을 확인해요`
- `팀 정보를 저장했어요`
- `로그인하면 이어서 진행할 수 있어요`

피해야 할 예:

- `사용자 입력값을 기반으로 프로세스가 진행됩니다`
- `다양한 기능을 통합적으로 제공합니다`
- `플로우 완료 후 다음 단계로 전환됩니다`

### 3. 구현 언어 줄이기

시스템이 어떻게 만들어졌는지 설명하는 말보다, 사용자가 무엇을 경험하는지 설명하는 말을 씁니다.

일반 UI 문구에서 피할 표현:

- `구현`
- `API`
- `mock`
- `payload`
- `상태 코드`
- `검증 로직`
- `원천 데이터`

사용자에게 가까운 표현:

- `준비됨`
- `연결`
- `샘플`
- `요청 내용`
- `오류`
- `확인`
- `활동 기록`

개발자가 제품을 이해하는 데 필요한 기술 용어는 쓸 수 있습니다. 다만 화면 맥락 안에서 의도적으로 써야 합니다.

### 4. 한 문장에는 하나의 생각만 담기

긴 UI 문장은 쉽게 생성된 설명처럼 보입니다. 나누거나 줄입니다.

좋은 예:

- `역할과 기술 스택을 등록하면 함께할 팀을 찾아요.`
- `팀이 만들어지면 체크리스트와 GitHub 활동을 한곳에서 볼 수 있어요.`

피해야 할 예:

- `사용자가 입력한 역할과 기술 스택 정보를 바탕으로 적합한 팀 매칭 경험을 제공하며 이후 팀 운영에 필요한 여러 기능을 통합적으로 사용할 수 있습니다.`

### 5. 제품 상태를 정확히 말하기

현재 제품이 지원하지 않는 일을 가능한 것처럼 쓰지 않습니다. 기능이 준비 중이거나 설정이 필요하다면 그대로 말합니다.

좋은 예:

- `준비하고 있어요`
- `연결하면 볼 수 있어요`
- `아직 등록된 내용이 없어요`

피해야 할 예:

- `자동으로 최적화해요`
- `모든 진행 상황을 완벽하게 분석해요`
- 설정이 필요한데 `바로 확인할 수 있어요`

## 주요 용어

같은 개념은 화면이 달라도 같은 단어로 씁니다.

| 의미 | 권장 표현 | 메모 |
| --- | --- | --- |
| 팀 작업 공간 | `팀 스페이스` | 팀 작업이 이어지는 주요 공간입니다. |
| 팀 찾기/결성 | `매칭` | 팀을 찾거나 만드는 흐름 전반에 씁니다. |
| 매칭 제출/관리 | `매칭 요청` | 실제로 요청을 제출하거나 관리할 때 씁니다. |
| 진행 상태 | `진행 상황`, `진척` | 사용자가 이해하기 쉬운 상태 표현을 우선합니다. |
| 작업 항목 | `할 일`, `체크리스트` | 화면 구조에 맞춰 선택합니다. |
| 개발 기록 | `GitHub 활동`, `개발 기록` | 데이터 출처보다 사용자 가치를 먼저 말합니다. |
| 예시 경험 | `샘플` | 실제 샘플이 보일 때만 씁니다. |

## 버튼과 링크

버튼과 링크는 짧고 행동 중심으로 씁니다.

좋은 패턴:

- `시작하기`
- `확인하기`
- `저장하기`
- `요청하기`
- `연결하기`
- `다시 시도하기`

기준:

- 주요 버튼은 핵심 행동을 말합니다.
- 보조 버튼은 이동할 화면이나 대체 행동을 말합니다.
- 링크 문구는 단독으로 읽어도 의미가 있어야 합니다.
- 결과가 분명하지 않을 때는 `다음`, `확인`, `이동`처럼 넓은 표현을 피합니다.

## 빈 상태

좋은 빈 상태는 세 가지를 답합니다.

- 무엇이 없는가?
- 왜 중요한가?
- 다음에 무엇을 할 수 있는가?

예시:

- `아직 받은 제안이 없어요. 역할과 기술 스택을 등록하면 함께할 팀을 찾기 시작해요.`
- `등록된 체크리스트가 없어요. 첫 할 일을 추가해 팀 진행 상황을 맞춰보세요.`
- `연결된 활동이 없어요. 저장소를 연결하면 개발 기록을 팀 스페이스에서 볼 수 있어요.`

## 오류와 검증 문구

쉬운 말로 씁니다. 먼저 문제를 말하고, 필요하면 다음 행동을 안내합니다.

좋은 예:

- `이메일 형식이 올바르지 않아요.`
- `비밀번호를 입력해 주세요.`
- `요청을 처리하지 못했어요. 잠시 뒤 다시 시도해 주세요.`
- `로그인이 필요해요. 계속하려면 먼저 로그인해 주세요.`

피해야 할 예:

- `Validation failed.`
- `요청 실패: 500`
- `인증 토큰이 존재하지 않습니다.`

## 점검 목록

문구 변경을 마무리하기 전에 확인합니다.

- 실제 경로, 상태, 제공되는 기능과 문구가 맞는가?
- 사용자가 다음 결정을 하는 데 도움이 되는가?
- 내부 구현 용어를 줄였거나 쓴 이유가 분명한가?
- 톤이 차분하고 자연스러운가?
- 좁은 UI와 모바일에서도 문장이 잘 들어가는가?
- 가까운 화면에서 같은 개념을 같은 단어로 부르는가?
